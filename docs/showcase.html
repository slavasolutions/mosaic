Mosaic 0.8 — drafting toward 1.0

A folder shape
for websites.

Put your content in a folder. Any tool that speaks Mosaic turns it into a website. Different tools, same folder, same site. No database. No platform lock-in. The folder outlives the tool.

The shape

The big idea, in one picture.

A Mosaic site is a directory with five kinds of things in it. Hover or click an entry to see what it does. That is the entire surface — if you can read these, you can build a Mosaic engine.

The spine

Seventeen truths.

Every rule in the spec traces back to one of these. If a truth changes, the spec changes. If you only read one thing, read this list.

The folder is the website.

Mosaic describes a directory on disk. The filesystem is the source of truth. Engines read it; nothing else is canonical.

A site has five things on disk.

A manifest, routed pages, collections of records, singletons at the root, and (optionally) images. Anything else is ignored.

A record is markdown, JSON, or both.

One file, a sidecar pair, or a folder containing index.*. Same rules everywhere.

Frontmatter is forbidden.

JSON is the only structured channel. A third channel forces a precedence rule that's never clean.

Refs come in four forms.

ref:, asset:, ./, and @selector suffixes. Engines detect refs by scanning string values — not by consulting a schema.

Addresses are flat.

A ref's address is either a singleton name or <collection>/<slug>, split on the first /. No deeper paths.

Refs are stubs in the index.

No eager inlining. Cycles are free. Consumers follow stubs on demand.

Validation has three levels.

Structural (no index), drift (index plus report), warning (cosmetic). Every diagnostic carries a stable error code.

Routing is declared by pages, not collections.

A collection becomes URLs only when a page mounts it with collection-list. routes: false mounts without claiming URLs.

mosaic.json is a manifest, not a snapshot.

It declares what the site should be — identity, types, collections, singletons, redirects, tokens. The folder is what currently is.

Writers preserve unknown fields.

Any tool that writes a Mosaic file MUST round-trip unknown fields verbatim. No silent data loss.

Engines run native or embedded.

Native engines use Mosaic's route table to serve URLs. Embedded engines treat the index as a queryable store while Astro, Next, or SvelteKit owns routing. Both are first-class.

Title precedence: JSON > markdown H1 > slug.

Required-title validation runs against the resolved title, not the raw JSON field.

The spec version is declared in mosaic.json#version.

Engines MAY refuse to read sites whose version they do not support.

Design tokens are content.

Stored as a root singleton, shaped per the W3C Design Tokens Community Group format. Mosaic defines where tokens live, not which colors a site should use.

Redirects are first-class.

Declared in the manifest. Map old URL to new URL. Loops are structural errors. Renderers emit them as 301 hints; routing engines apply them server-side.

Home is /.

pages/index.{md,json} mints the home route. The slug home is reserved at the top level. Engines auto-alias /home to / so authors and visitors never collide on the spelling.

A real example

Show, don't tell.

This is a slice of examples/hromada-community/, the canonical example site. Every file shown here is checked into the repo, validated end-to-end, and rendered by the reference engine.

mosaic.json 
{
  "$schema": "https://mosaic.dev/schemas/0.8.json",
  "version": "0.8",
  "site": {
    "name": "Hromada Community",
    "locale": "en-CA",
    "url": "https://hromada.example.org"
  },
  "types": {
    "TeamMember": {
      "fields": {
        "title": { "type": "string", "required": true },
        "role":  { "type": "string", "required": true },
        "photo": { "type": "asset" },
        "bio":   { "type": "markdown" }
      }
    }
  },
  "collections": {
    "team": { "type": "TeamMember", "defaultMount": "/team" },
    "news": { "type": "NewsItem",   "defaultMount": "/news" }
  },
  "singletons": {
    "site":   { "type": "SiteConfig" },
    "tokens": { "type": "DesignTokens" }
  },
  "redirects": [
    { "from": "/news/launch", "to": "/news/2025-03-12-launch", "status": 301 }
  ]
}
Manifest

The manifest declares what the site should be.

Not a snapshot. Not a route table. The manifest names the site, declares its types and collections, lists its singletons, and pins redirects. The folder shows what currently is; the manifest shows what the site should be.

  • version — engines may refuse versions they don't speak.
  • types — record shapes referenced by collections.
  • collections — declare a collection and its default mount path.
  • singletons — root-level records like site.json, tokens.json.
  • redirects — first-class. The old URL stays alive.
pages/index.json 
{
  "title": "Hromada Community",
  "sections": [
    {
      "type": "hero",
      "headline": "Belonging, built together",
      "subhead": "A community space for Ukrainian-Canadians in the GTA.",
      "image": "asset:images/hero.jpg",
      "cta": { "label": "Get involved", "url": "/contact" }
    },
    { "type": "prose", "from": "./index.md" },
    {
      "type": "feature-grid",
      "items": [
        { "title": "Settlement support", "ref": "ref:services/settlement" },
        { "title": "Language classes",   "ref": "ref:services/language" },
        { "title": "Cultural events",    "ref": "ref:services/culture" }
      ]
    },
    {
      "type": "collection-list",
      "from": "collections/news",
      "sort": "date desc",
      "limit": 3,
      "routes": false
    }
  ]
}
Home page

The home page mounts content from elsewhere in the folder.

pages/index.json becomes /. Its sections pull in three other pieces of content: an asset, a markdown sibling, and a slice of the news collection. Three of the four ref forms are on this page.

  • asset:images/hero.jpg — a binary asset.
  • ./index.md — markdown sibling next to this JSON.
  • ref:services/settlement — a record in the services collection.
  • routes: false — list the latest news here, but don't claim /news/* URLs from this page.
collections/team/anna/index.json 
{
  "title": "Anna Kovalenko",
  "role": "Executive Director",
  "email": "anna@hromada.example.org",
  "photo": "./photo.jpg",
  "colleagues": ["ref:team/ben", "ref:team/maria"],
  "bio": "./bio.md"
}
collections/team/anna/bio.md 
# Anna Kovalenko

work supporting newly arrived families across the GTA.

Collection record

One person, one folder, two files.

A record can be a single file, a sidecar pair, or a folder. Anna is a folder — the JSON holds her structured fields, the markdown holds her bio, and a co-located photo lives in the same directory. The folder shape lets assets sit next to the record they belong to.

  • title — required. Drives the resolved title (JSON wins over H1 wins over slug).
  • colleagues — an array of cross-collection refs. Cycles are fine: refs are stubs.
  • ./bio.md — relative ref to a sibling inside the same record folder.

    "muted":      { "$value": "#6b6b6b", "$type": "color" }
  },
  "font": {
    "body":      { "$value": "system-ui, sans-serif",   "$type": "fontFamily" },
    "heading":   { "$value": "Georgia, 'Times', serif", "$type": "fontFamily" },
    "size-base": { "$value": "16px", "$type": "dimension" }
  },

Design tokens

Tokens are content too.

Colors, fonts, and spacing live in a root singleton named tokens.json, shaped per the W3C Design Tokens Community Group format. Renderers convert tokens to CSS variables. Pages and components address tokens with ref:tokens@color.accent.

  • Mosaic owns the address of tokens, not the values.
  • Editors can rewrite the file without losing unknown keys (truth 11).
  • The same file works in any engine that understands Mosaic.

Addressing

Four ref forms. That's the grammar.

When one piece of content points to another, it uses a ref. Engines detect refs by scanning string values for these prefixes — the schema does not decide. Every link inside a Mosaic site looks like one of these.

ref:<name> — a singleton

The name maps to a singleton at the root: site.json, site.md, or both.

ref:<collection>/<slug> — a record

Split on the first /: collection name, then slug. Addresses are flat — never nested.

asset:<path> — a binary asset

Always under images/. The optional images/manifest.json carries width, height, alt text.

ref:<addr>@<selector> — a field

The @selector suffix picks one path inside the resolved value — dotted, no array indexing.

Diagnostics

Three validation levels. Stable error codes.

Every problem the validator finds carries a level and a stable code. Tools match on codes; messages may change between versions. CI fails on structural; drift is reported but the index still builds.

Structural

Site can't be built.

The folder violates the spec at a level the engine can't recover from. No index is produced. CI must fail.

mosaic.home.reserved
pages/home.md uses the reserved home slug.
Drift

Site builds. But it's inconsistent.

The index is produced AND the issue is reported. The manifest and the folder disagree about something the engine can paper over.

mosaic.ref.unresolved
ref:team/dana points at a slug that doesn't exist.
Warning

Cosmetic. Engine notices.

The site is fine. Something is unusual or duplicated. Reporting is optional — useful for editors, ignored by CI.

mosaic.title.dead-h1
JSON title wins; the markdown H1 becomes dead text.

End to end

Author. Validate. Index. Consume.

Four independent steps, in a one-way pipeline. Authors don't care which renderer ships next. Renderers don't care which editor produced the folder. Mosaic is the shared shape between them.

AuthorFiles in any text editor. Git. AI agents. A future CMS that speaks Mosaic.
ValidateWalk the tree. Emit diagnostics with stable codes. Block on structural.
IndexProduce one JSON document — pages, records, refs as stubs, routes.
ConsumerReference renderer, Astro adapter, Next adapter, downstream editors.

Fit

What you can build with it.

Anything that fits in a folder. Mosaic doesn't care what the content means — just that it sits in the right shape.

Personal blog

One post per markdown file under collections/posts/. Tags as a sidecar JSON. Three-line manifest. Done.

Documentation site

Folder-shape records nest cleanly. A page mounts a docs collection; markdown becomes pages; JSON sidecars carry metadata.

Portfolio

One folder per project. JSON for fields, markdown for the writeup, photos co-located. Refs link related work.

Community site

News, team, events, services — all collections with default mounts. The canonical example in this repo does exactly this.

Small e-commerce catalog

Products as a collection. Sku, price, gallery via assets. Cart and checkout live outside Mosaic; content stays in the folder.

AI-edited content store

An agent reads the folder, edits files, writes back. Unknown-field preservation keeps engine-specific data intact across writes.

Real code

Implementations shipping in this repo.

Each consumer demonstrates a different way to read a Mosaic folder. All of them work against the same canonical example.

VAL
tools/validate · Node · zero deps

Reference validator

Walks the folder, emits diagnostics with stable codes, passes the conformance suite. Copy patterns or use directly in CI.

RND
tools/render · wireframe HTML

Reference renderer

Turns an index into a basic static site. Tokens become CSS variables. Useful as a proof and as a starting point for richer renderers.

AST
tools/astro-adapter · embedded engine

Astro adapter

Drop a Mosaic folder into an Astro project. The adapter exposes the index as a content collection; Astro owns the routing.

CMS
downstream · separate project

Clear CMS

A browser-based editor that reads and writes Mosaic folders. Round-trips unknown fields verbatim (truth 11) so other engines stay safe.
