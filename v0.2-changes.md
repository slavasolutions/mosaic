# Mosaic spec v0.2 — draft changes

This document summarises the additive, non-breaking sections introduced in the `v0.2-draft` branch. None of v0.1's normative content has been altered.

## Summary

Four additions, each marked `(v0.2 DRAFT)` in `spec.md` for reviewer visibility.

### 1. `LayoutSpec` body — §4.6a + §5.1a

v0.1 declared `"layouts": { "<name>": <LayoutSpec>, ... }?` in `mosaic.json` but left `LayoutSpec` undefined. v0.2 fleshes it out as a responsive CSS-Grid-shaped template: a `grid` object (columns + gap token), an `areas` map keyed by breakpoint (using `grid-template-areas` row syntax), an optional `breakpoints` map (key → min viewport width in pixels), and `sectionDefaults.area` for unassigned sections. Section instances opt into a named area via the new optional `sectionLayout: { area, span? }` field on `SectionInstance` (§5.1a). Implementable in browser-native CSS Grid; non-CSS renderers MAY approximate as a vertical stack. Level 1 readers MAY ignore `layouts` entirely.

### 2. `tokenOverrides` cascade — §4.2a + §5.2 + §5.1 SectionInstance field

Tokens were site-level only in v0.1. v0.2 adds a scope cascade: site → page → section. (Slot-level was considered and deferred — see open question below.) `tokenOverrides` is a two-level map identical in shape to `mosaic.json#tokens` and lives optionally on page records (§5.2) and on `SectionInstance` (§5.1). Resolution is last-write-wins along the cascade. Overrides MUST only override token NAMES that already exist at the site level — introducing a new name via override is invalid (readers ignore + warn). Renderers SHOULD emit scoped CSS custom property declarations on the page/section wrapper. Token VALUE changes do NOT bump `schemaVersion`. Support is OPTIONAL for Level 1.

### 3. `freeform` canonical block type — §10b

A canonical block type for absolute-positioned design-heavy content (covers, editorial inserts, posters, art-directed callouts). Declared aspect-ratio canvas, list of `freeformItem` structs (`kind` ∈ `"text" | "asset" | "shape"`, plus `position: { x, y, w, h, z?, rotation?, units? }`). Default coordinate basis is percent of canvas; `units: "px"` opts into absolute pixels. Renderers SHOULD use `position: absolute` against a `position: relative` canvas. Spec explicitly documents the portability tradeoff: `freeform` is the LEAST portable canonical block across surfaces — headless consumers (§6.1) SHOULD include it in their skip set unless they specifically support it. Level 1 readers MAY treat it as unknown.

### 4. Consumer capability declaration — silent skip — §6.1

v0.1 §6 says invalid sections surface via a warnings channel, and unknown block types render as `<missing template>`. That's correct for engine-default renderers but wrong for headless consumers (custom renderers, mobile, AI, RSS, search indexers) that are partial projections of a document. v0.2 adds §6.1: a consumer MAY declare a supported block types set and MUST silently drop unsupported blocks from the render tree (no fallback, no placeholder, no warning for the unsupported `blockType` itself — separate from §6 validation, which still applies to supported sections). Canonical pattern: a `SKIP_BLOCKS` set on the consumer side. Pass-through serialisers (Level 2 writers) SHOULD preserve unsupported sections so other consumers still see them — silent skip applies to rendering, not storage. `mosaic.json` MUST NOT declare per-consumer skip rules. This contract is OPTIONAL for engine-default renderers.

## Open questions

- **Slot-level `tokenOverrides`** — the cascade in §4.2a stops at section. Slot-level was considered (e.g. "make this one button's `color.accent` red") but rejected for now: it bloats the slot value schema (every slot would need an optional override sidecar), it conflicts with translatable slot serialisation, and the same effect is already achievable by overriding at the section that holds the slot. Keep section as the narrowest scope until a concrete authoring need emerges.
- **`freeform` units mixing** — current spec lets each `freeformItem` declare its own `units` independently. Likely fine, but the visual outcome of mixing percent and px items in the same canvas is undefined under arbitrary aspect rescaling. Open question whether `units` should be a canvas-level field instead of per-item.
- **`LayoutSpec.areas` per-breakpoint row-count mismatch** — spec says row counts MAY differ across breakpoints; renderers will then have rows that "appear" at large widths only. Reasonable for CSS Grid but raises questions for non-CSS approximators about how to order them in the fallback stack.
- **`sectionLayout.span` semantics** — declared as a column span integer, but with named areas (which already control geometry) `span` is mostly redundant. Kept as an optional refinement, but may be removable if no real authoring need surfaces.
- **`freeform` and headless skip default** — §10b.6 says headless consumers SHOULD include `freeform` in their skip set. Worth considering whether the spec should go further and RECOMMEND it as the default behavior in the §6.1 canonical pattern (i.e. ship `freeform` in the default skip set for headless consumer reference implementations).
