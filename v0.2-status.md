# v0.2 status matrix

## What this doc is

A row-by-row map of every part of the Mosaic spec, showing what existed in v0.1, what v0.2 added, what landed today after the peer review, and what's still on the to-do list for v1.0. Written for non-developers — if a term needs more depth you can google it. Tables are split into small groups so they stay readable in any markdown viewer.

---

## Two quick answers first

**Are we making new files in the repo?** No. Two files were edited today: `spec.md` and `v0.2-changes.md`. `REVIEW.md` was already there. This file (`v0.2-status.md`) is the only new one, and it's not part of the spec — it's a reading aid. None of these are committed yet.

**Why did the spec get bigger?** v0.2 started as 4 small additions. The peer review surfaced 6 more real gaps that needed closing (mostly things the original spec referenced but never defined). So today's spec is ~200 lines longer than yesterday's. That's the review doing its job — not scope creep.

| §4.1 `meta` | Site name + `schemaVersion` (bumps on breaking change) + optional domain. | Defined. | — | — | — |
| §4.2 `tokens` | Design tokens (colors, fonts, spacing). Renderers turn these into CSS variables. | Defined. | — | — | "Common groups" paragraph (color/font/size/etc.) reads as filler — should move to an appendix. |
| §4.2a **`tokenOverrides` cascade** | Pages and sections can override site tokens for a narrower scope. ## Group 4 — Validation and reference resolution (§6–§7)

Validation says what makes a section legal. Reference resolution explains how the strings inside slot values (`asset:foo.jpg`, `ref:blog/post`, `block:sha256-...`) get resolved into actual content.

| ID | What it is | v0.1 | v0.2 | Current | Planned |
|----|------------|------|------|---------|---------|
| **§6 Validation** | When is a section "invalid" (wrong block type, missing required slot, broken ref, list outside min/max). Invalid sections MUST surface via warnings; MUST NOT crash. | Defined. | — | — | Minor RFC-2119 wording cleanup (bullets after a MUST should themselves carry MUST/MAY). |
| §6.1 **Silent skip** | Headless consumers (RSS feeds, AI tools, search indexers) each support only a subset of block types. They declare a supported set and silently drop the rest — no `<missing template>` placeholder. Storage still preserves the dropped sections so other consumers see them. | Did NOT exist. The default was `<missing template>` for unknowns, which is wrong for headless. | NEW. The reviewer called this "the most novel contract in v0.2" — they're not aware of prior art for it in a format spec. | — | Order-of-operations: when a section is invalid AND in the skip set, do other validation warnings still surface? Spec is silent. |
| **§7 Reference resolution** | The four ref shapes (`asset:`, `ref:`, `block:sha256-`, and the Markdown body form `asset:content/*.md`) and how a resolver turns them into objects. | Defined as four shapes by example only — no formal grammar. Reviewer called this the spec's #1 highest-leverage gap. | — | Now backed by new normative **Appendix C** with a full grammar (ABNF) and 5 normalisation rules. §1 Level 1 now points to Appendix C. | — |
| §7.1 `asset:<path>` | References a file in `assets/`. Resolves to `{ kind, path, sha256, url }`. | Defined. | — | — | — |
| §7.2 `ref:<collection>/<id>` | References a record in a collection (like a blog post). | Defined. | — | — | — |
| §7.3 `block:sha256-<hash>` | Engine-internal content-addressed block reference. | Defined but described via two undefined terms ("Tier 2 storage", "hybrid storage"). Implementers couldn't tell what was required. | — | Both undefined terms removed. Replaced with concrete contract: content-addressed block storage is engine-OPTIONAL; refs without backing storage resolve to `{ content: null }` and that's fully conformant. | The shape of the `content` field is still "engine-defined." Acceptable for now; worth revisiting (M5 in the review). |
| §7.4 `asset:content/*.md` | Special form: inline a Markdown file body. | Defined as "special form of §7.1" — vague. | — | Appendix C C.2 clarifies: syntactically same as §7.1, behaviorally the renderer detects `.md` and inlines as CommonMark + GFM. | — |
| §7.5 Circular refs | If A points to B and B points back to A, the resolver MUST detect the cycle and not infinite-loop. | Defined. | — | — | — |

## Deferred to v1.0 — the full deferred list

Everything below was raised by the peer review or by v0.2's own open questions, and consciously NOT done in this draft. Each is a real issue, but v0.2 ships additive-only — these need either breaking changes or larger design conversations.

**From the peer review (13 items):**

- **C6** — empty `blockTypes` + non-empty pages = every section invalid. Spec should explicitly handle zero-content valid sites.
- **C7** — if `assets/` exists but `manifest.json` doesn't, the spec contradicts itself (§3 says manifest REQUIRED if `assets/` exists; §8 says resolvers SHOULD tolerate manifest absence).
- **M1** — `richtext` value has no built-in marker for markdown vs portable-text. Parser must consult the schema.
- **M2** — `i18n.fallback` semantics. Per-slot? Per-record? Per-route? Field-level missing-key behavior?
- **M3** — `globalsOverride` example shows duplicate JSON keys (illegal). Needs rewriting as a typed union.
- **M4** — globals ordering when multiple share a position with no `above`/`below`. Unspecified = non-portable.
- **M5** — `block:` ref `content` shape is still partly engine-defined. Spec should say more, even if "implementations differ."
- **M6** — `freeform` percent + px units mixing under canvas rescale. Visual outcome undefined.
- **M7** — `tokenOverrides` partial-shadow case (override has some valid + some invalid names).
- **M8** — `state: "draft"` section interaction with `sectionLayout`, `publishedHash`, `before-section:<n>`.
- **Validation order with §6.1 skip set** — when a section is invalid AND skipped, what surfaces?
- **JSON Schema for `mosaic.json` itself** — formal validator schema. Promised in v0.2 plan, not delivered.
- **Schema migration** — what does a renderer do when it reads a higher `schemaVersion` than it understands?

**From v0.2's own open questions (5 items):**

- Slot-level `tokenOverrides` (4th tier — site/page/section/slot). Deferred because it bloats every slot value.
- `freeform` units mixing — likely promote `units` to canvas-level.
- `LayoutSpec.areas` per-breakpoint row count differences — non-CSS fallback ordering.
- `sectionLayout.span` semantics with named areas — possibly redundant, removable.
- `freeform` default-skip recommendation — promote SHOULD to RECOMMENDED for headless.

**The big unfalsifiable one:**

- **Level 3 concurrent-merge semantics.** Spec says L3 MUST "preserve concurrent edits without data loss" but never defines what counts as preservation. Until that's defined, no one can prove a Level 3 implementation is conformant.

**Minor nits (~15 items):**

These are formatting/wording cleanups (RFC-2119 hygiene, `.clear` vs `.loro` consistency, the `clear.json` synonym baking vendor into spec, pipe-vs-backtick alt syntax inconsistency, etc.). Listed in `REVIEW.md` under "Minor issues / nits."
