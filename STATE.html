Philosophy — 8 foundational claims

The spec derives from these. Plain language, no MUST/MAY.

1. The folder is the website

The filesystem is the source of truth; nothing else is canonical.

2. mosaic.json is the manifest

One special file at the root identifies the folder as a Mosaic site and declares its shape — types, collections, redirects, tokens.

3. Files are records, folders are collections open

Records carry data in JSON; markdown and other content is referenced from them. Folders group records.

Open: does "all root folders = collections" hold up?

The .mosaic zip layout showed an apparent inversion: images have optional sidecars, collection records have required JSON + optional sidecars.

  • Option A (recommended): all root folders are collections. The "inversion" is the SAME rule applied to different content types — the file (binary or JSON) IS the record; the JSON sidecar is optional metadata.
  • Option B: split content types — pages/, images/ reserved with own conventions.

My pick: A. Keeps universal composition. The "inversion" is cosmetic, not structural.

4. JSON is the only structured channel open

Markdown is prose; structured fields live in JSON.

Open: frontmatter support?

  • Option A: Spec stays silent. Engines may parse frontmatter as extension; not portable.
  • Option B: Hard ban. Migrator tool extracts frontmatter to JSON.
  • Option C (your latest): "Not native now, future MIP."

My pick: A + C combined. Spec silent for 0.9. Engines may parse; tools may convert. Future MIP placeholder filed.

5. Refs link records mostly locked

One ref: prefix, three anchors: cascade (default), / absolute, ./ explicit relative. Selectors restored: ref:tokens@color.accent works.

  • locked Unified x- namespace marker everywhere (fields AND sidecar filenames)
  • open Cascade + locale interaction order — needs concrete examples; defer
  • critical Ref identity (rename safety) — see blind spot

6. Routing is declared, not derived

Pages-collection paths become URLs; other collections route only when a page mounts them.

7. Deeper scope wins (cascade)

Cascade walks parent chain. Objects merge field-by-field; arrays replace whole.

You asked: what's the problem? / exists as escape hatch.

Right — escape hatches handle the "I want exact path" case. The remaining concern was silent debugging hell (closer-wins shadowing breaks pages invisibly). Mitigation: mosaic.cascade.shadow warning fires whenever cascade resolves non-root. Authors see the cascade graph; tooling surfaces shadowing.

8. Forward-safe

Engines decide URLs; writers preserve unknown fields; extensions use x- namespace (fields and sidecars). On-disk conflict emits MOS-CONFLICT markers.

Architecture decisions (locked)

A1. .mosaic = single distribution format (zip-based)

EPUB-style zip with .mosaic extension. Tools open as zip; users see one file. No light/heavy split — one canonical format.

A2. JSON-only structural index

Every record's structural entry is JSON. MD = content referenced from it (optional auto-link if same-slug). Bare MD is engine charity, warning emitted.

A3. Cross-engine round-trip testing

Every conformance fixture is validated TWO ways:

  1. Direct: folder → emit index → compare to expected
  2. Round-trip: folder → zip to .mosaic → unzip → emit index → compare

Indexes must match. Catches: zip metadata leaks, path normalization, locale parsing, file ordering, Unicode normalization.

A4. Cascade lookup walks parent chain (no carveouts)

Page records and globals coexist; cascade naturally never crosses into sibling collections.

A5. Sidecar override = cascade applied to globals

Same mechanism. No separate "override" spec rule needed.

A6. Unified x- namespace marker

Fields AND sidecar filenames use x-. x-astro.localized for fields, header.x-clearcms.json for sidecars.

A7. @ selector restored to ref grammar

DTCG tokens need it: ref:tokens@color.accent.

Open Q3: Core in spec or separate?

My pick: separate. Spec defines what's a valid Mosaic site. Core is one reference implementation. Other engines (Go, Rust, Python) should be free to reimplement without vendoring our JS code.

Open Q4: MosaicDB direction

You clarified: not R2-only. General engine to bolt onto any filesystem.

  • @mosaic/db = a stateful engine holding a Mosaic site; exposes read/write/query API
  • Backend agnostic — local FS, R2, S3, any path-keyed storage
  • Built on @mosaic/core
  • Could BE the viewer/editor's backend (Studio = UI on top of @mosaic/db)
  • Clear could compose @mosaic/db + custom UX

Post-1.0 territory. File in V1.md futures.

🔴 Biggest blind spot: ref identity & rename safety

Every adjacent project had to answer "what does a ref actually point at, and how do I resolve it when the target moves or the folder is copied somewhere else?"

  • OCI: content-addressed digests
  • BagIt: checksums
  • STAC: relative + absolute URIs with rel types
  • Frictionless: path with explicit relative/remote rules
  • Nix Flakes: lock-pinned input refs

Cascade answers MERGE semantics; not IDENTITY. If Mosaic's refs are folder paths today, the spec is one rename away from breaking. No story yet for:

  • Refs across two Mosaic folders on the same disk
  • Refs in a planned R2 cloud variant where folders are distributed
  • What happens when team/anna renames to team/anna-k — every ref silently breaks

Open Q10 — options for ref identity

  • A: Refs stay path-based; renames break refs; mosaic.ref.unresolved catches via validation
  • B (recommended): Add optional stable $id to records; refs can point at $id OR path; engines resolve either; rename-safe
  • C: Full content addressing (OCI-style hash); immutable but verbose refs

My pick: B. Path-based default for ergonomics; identity-based escape hatch for stability. Worth a MIP-0016.

Open answers to questions raised this session

What is BSON, and is it useful?

BSON = Binary JSON. How MongoDB stores documents. Faster to parse, includes types JSON doesn't (Date, ObjectId, Binary). Not useful for Mosaic spec — design is filesystem-native + human-readable. Binary formats break "open in any text editor." Engines that want fast in-memory representation parse JSON to whatever internal shape they want; implementation choice.

Should you NEED CLI to make the index?

No. Index is engine OUTPUT, not spec requirement. Engines emit it. CLI is one way; hand-authoring possible. Spec documents the index shape (§7); any tool can emit/consume it.

Mermaid sucks for visualization — what to use?

Drop it for decision trees. THIS document is the structured plan. Each section is a node; status emoji shows state; cross-references inline by section. Plain text/HTML scales and renders everywhere. The visual hierarchy in this HTML version (cards, badges, TOC) does the work mermaid couldn't.
