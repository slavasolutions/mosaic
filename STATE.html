Principles — 3 foundational claims

The spec derives from these. Plain language, no MUST/MAY. See PRINCIPLES.md for the canonical file.

1. The folder is the website

The filesystem is the source of truth. Files are records (.json = structured, .md = prose, anything else = binary/non-text content). Folders group records into collections.

2. Refs link records

One ref: prefix with three anchoring modes: cascade lookup (default — walks the parent chain outward), / for absolute from root, ./ for explicit relative. Deeper-wins cascade plus deep merge: same-named records placed deeper override shallower ones; objects merge by field, arrays replace whole. Cycles are free.

3. Forward-safe

Engines decide URLs. Writers preserve unknown fields. Extensions use the x- marker (fields as x-<ns>.<key>; sidecar files as <slug>.x-<ns>.json). Types MAY declare an optional @type field for schema.org alignment — see Schema.org section below.

Reduced from 8 truths to 3 principles. Everything cut became derived rules in SPEC.md (file extension determines role; mosaic.json is the manifest; routing is declared by the pages collection; validation has tiered severity). Same content, fewer claims to memorize.

Schema.org alignment — optional, type-level

Types in mosaic.json#types MAY declare an @type field naming a schema.org type. Example:

"types": {
  "TeamMember": { "@type": "Person",  "fields": {...} },
  "NewsItem":   { "@type": "Article", "fields": {...} }
}

What this gives you

  • Engines that emit JSON-LD use @type to mark records semantically. Free SEO + AI discoverability.
  • Engines that don't care just ignore the field.
  • Authors who don't care leave it off; nothing breaks.

What we explicitly did NOT do

  • No "profile" concept. No site-level alignment modes. Profiles add validator complexity and force authors to pick a tier; we dropped them.
  • No required schema.org field names. Authors MAY align field names by convention (name for Person, headline for Article); spec doesn't require it.

Three layers exist conceptually but aren't formalized:

  1. Default — no @type, engines emit plain HTML.
  2. Recommended — types declare @type. Engines that support JSON-LD emit it. SEO benefit, low cost.
  3. Advanced — authors align field names too. Schema.org-aware tools get full structured data.

The optional @type field on type declarations is the spec's only hook. Everything else is engine and author choice.

Architecture decisions (locked)

A1. .mosaic = single distribution format (zip-based)

EPUB-style zip with .mosaic extension. One canonical format.

A2. JSON-only structural index

Every record's structural entry is JSON. MD = content referenced from it (optional auto-link if same-slug). Bare MD is engine charity, warning emitted.

A3. Cross-engine round-trip testing

Every conformance fixture validated direct AND zip+unzip → emit index → compare. Catches zip metadata leaks, path normalization, locale parsing, file ordering, Unicode normalization.

A4. Cascade walks parent chain (no carveouts)

Pages and globals coexist; cascade naturally never crosses into sibling collections.

A5. Sidecar override = cascade applied to globals

Same mechanism. No separate "override" spec rule needed.

A6. Unified x- namespace marker

Fields AND sidecar filenames use x-. x-astro.localized for fields, header.x-clearcms.json for sidecars.

A7. @ selector in ref grammar

DTCG tokens need it: ref:tokens@color.accent.

A8. JSON Schema for record field types

mosaic.json#types uses JSON Schema instead of bespoke type shape. Free tooling, less reinvention.

🔴 Biggest blind spot: ref identity & rename safety

Every adjacent project solved "what does a ref point at when the target moves?" — OCI: content-addressed digests; BagIt: checksums; STAC: relative+absolute URIs with rel types; Frictionless: explicit path rules; Nix: lock-pinned input refs.

Cascade answers merge semantics, not identity. If refs are folder paths today, the spec is one rename away from breaking.

Q10 options

  • A: Refs path-based; renames break refs; mosaic.ref.unresolved catches via validation
  • B (recommended): Add optional stable $id to records; refs can point at $id OR path; engines resolve either
  • C: Full content addressing (OCI-style hash); immutable but verbose refs

Pick: B. Worth a MIP-0016. Path-based default for ergonomics; identity-based escape hatch.
